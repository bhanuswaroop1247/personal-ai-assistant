# PARA Implementation in Research Workflow

**Type:** Knowledge Reference | **Domain:** Productivity Systems  
**Last Updated:** 2025

---

## What Is PARA?

PARA is an organizational system for digital information developed by Tiago Forte. It provides a universal structure applicable across all apps, tools, and devices:

- **P – Projects:** Active endeavors with a defined outcome and deadline.
- **A – Areas:** Ongoing responsibilities with no end date.
- **R – Resources:** Topics and interests of ongoing reference value.
- **A – Archives:** Inactive material from the other three categories.

The critical insight: **organize by actionability, not by topic.** A file about keratoconus belongs in Projects if actively used for a paper, in Resources if it's reference knowledge, and in Archives when the paper is submitted.

---

## PARA in a Research Context

### Projects (Active Research Work)
Each paper, experiment, or deliverable is a project:
- Keratoconus Severity Classification – Paper Draft
- Pupil Tracking Application – Development
- ML Interview Preparation

**Naming convention:** `[Topic] – [Deliverable/Status]`  
Each project folder should contain: a project note (goals, deadline, next action), working documents, data references.

### Areas (Research Responsibilities)
Things maintained continuously with no finish line:
- Academic skills development
- Lab equipment competency
- Research ethics / integrity
- Programming skills

### Resources (Reference Library)
Reusable knowledge, independent of any project:
- Machine Learning concepts
- Ophthalmology clinical references
- Statistical methods
- Programming patterns

**Key rule:** Notes in Resources should have no deadlines, no "to-dos" — pure knowledge.

### Archives
- Completed projects (submitted papers, finished coursework)
- Past job applications
- Old academic notes (semester-specific)

## Workflow: Note Movement

```
Capture → Inbox
    ↓
Is it actionable with a deadline?  → Projects
Is it an ongoing responsibility?   → Areas
Is it reference material?          → Resources
Is it done or inactive?            → Archives
```

The system's value compounds over time: you build a searchable, evergreen knowledge base that grows with your research career.

---

## Practical Rules for Researchers

1. **One active note per project** — the "project note" with goals, current status, next action.
2. **Resources are searchable, not browsable** — don't obsess over perfect organization; tagging and search compensate.
3. **Weekly review** — move items from Inbox, check if any Projects are actually inactive (→ Archive).
4. **Don't duplicate** — if a note belongs in Resources AND Projects, keep the live copy in Projects and link to it; move to Resources when the project ends.

---

## Common Mistakes

- **Too many Projects:** If you're not actively working on it this week, it's not a Project.
- **Using Resources as a TODO list:** Resources should contain knowledge, not action items.
- **Never Archiving:** Completed projects left in Projects create cognitive clutter. Archive aggressively.
